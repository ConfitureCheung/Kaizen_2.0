Django project anatomy for myportal

Visual flow diagram

How your Django app works together.

This graphic shows the path from browser request to rendered page, plus how your folders map to Django’s MVT pattern and tenant-aware project structure.

Browser User opens a URL like /buildings/.
Routing config/urls.py and app URLs choose the correct view.
Logic views.py checks login and filters by client access.
Data + HTML models.py fetches data, then templates render the response.

Request lifecycle

1. User

Browser request

A person clicks a menu item like Dashboard, Buildings, or Clients.

  • GET /buildings/
  • Cookie/session sent
  • Needs response HTML
route
2. Router

URL matching

config/urls.py forwards the request into core/urls.py or accounts/urls.py.

  • Top-level URL entry
  • App-specific URL map
  • Chooses view function
dispatch
3. View

Business logic

views.py checks login, permission scope, and what data should be shown.

  • login_required
  • allowed client IDs
  • prepare context
query
4. Model

Database layer

models.py defines tables like Client, Building, Group, and User relationships.

  • SQL via ORM
  • SQLite / PostgreSQL
  • Client isolation boundary
render
5. Template

HTML response

templates/*.html extends base.html and sends final page markup back to the browser.

  • Shared top bar
  • Sidebar + tab shell
  • Page-specific content

config/

Project control room: settings, root URLs, deployment entry points.

accounts/

Authentication and identity: custom user, provider profile, login/logout.

core/

Business domain: clients, buildings, groups, memberships, portal logic.

templates/

Presentation layer: base layout and page HTML files.

static/

Frontend assets: CSS styling and JavaScript interactions like the hamburger menu.

→ models.pyDjango MVT
Defines tables, relations, and ORM queries
ViewClassic MVC screen
→ templates/*.htmlDjango MVT
Renders HTML shown in the browser
ControllerClassic MVC logic
→ urls.py + views.pyDjango MVT
Routes requests and handles business logic

Tenant access rule

Provider / Superuser Can see and manage all clients, buildings, groups, and users.
Client user Can only see rows linked to their own client membership.
Core filter point views.py limits querysets using allowed client IDs before rendering templates.
Practical memory hook: URLs decide, views think, models store, templates show, and static files style/animate.
